            Observation: By itself, string length doesn't really affect performance.
        Test 3: L = 1 # strings = 100000

        Test 4: L = 100 # strings = 100000

        Test 5: L = 10000 # strings = 100000

    InsertInOrderSpeedTest
        Test 1: # strings = 1
            Results: ULLMap = 0.00, MyHashMap = 0.00, Java HashMap = 0.00
            Observation: More a sanity test than anything. All can take a string of length 1 instantly.
        Test 2: # strings = 10
             Results: ULLMap = 0.00, MyHashMap = 0.00, Java HashMap = 0.00
             Observation: Same result as Test 1.
        Test 3: # strings = 1000
             Results: ULLMap = 0.01, MyHashMap = 0.00, Java HashMap = 0.00
             Observation: ULLMap lags barely perceptively.
        Test 4: # strings = 1000000
             Results: ULLMap = Stack Overflow, MyHashMap = 0.39, Java HashMap = 0.30
             Observation: ULLMap can not handle storing many objects.
        Test 5: # strings = 1000000000
             Results: ULLMap = Stack Overflow, MyHashMap = OutOfMemoryError, Java HashMap = N/A
             Observation: Too many elements to store.
        Test 6: # strings = 10000000
             Results: ULLMap = Stack Overflow, MyHashMap = 3.42, Java HashMap = 1.99
             Observation: MyHashMap now lags behind HashMap by more than a second.
        Test 7: # strings = 15000000
             Results: ULLMap = Stack Overflow, MyHashMap = 7.60, Java HashMap = 3.26
             Observation: MyHashMap takes about twice as long as the Java HashMap.

        Hashmaps speed is so great compared to other data types where if one is looking for speed of adding and getting,
        the hashmap is better than the other data structures we have looked at in this course. The main downside would be space use
        as hashmaps need both references to the data stored and the buckets used while other data structures really need just
        the data references.
